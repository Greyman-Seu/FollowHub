# FollowHub 第一阶段可执行任务拆解

> 本文档用于把第一阶段 PRD 拆解为可以逐项执行、验收和串联的工程任务。当前阶段只做任务规划，不进入开发实现。

## 1. 任务拆解原则

- 先收敛边界，再接代码
- 先打通最小闭环，再扩来源和质量
- 先复用参考代码，再决定哪些能力重写
- 重点保证第一阶段目标：每天稳定生成一份可读日报网页

## 2. 第一阶段总目标

交付一个本地运行的 FollowHub 原型系统，能够：

- 按配置读取 X、公众号、arXiv 信息源
- 采集并标准化当天内容
- 生成重点摘要与全量线索
- 产出日报 HTML、索引页、归档页
- 记录状态并支持最近 7 天补跑

同时在架构上预留后续来源扩展能力，包括：

- 个人网站
- 博客页面
- 公司网页
- 新闻网站
- 视频网站
- 其他可持续监控来源

## 3. 任务分组

第一阶段建议拆成 8 组任务：

1. 参考代码梳理与接入决策
2. 项目骨架与目录约定
3. 配置系统
4. 数据模型与中间产物
5. 信息源接入
6. 摘要与筛选流程
7. 页面渲染与输出
8. 调度、状态、补跑与运行

## 4. 可执行任务清单

### 任务组 1：参考代码梳理与接入决策

#### Task 1.1 盘点参考代码

目标：

- 找到你现有的 X / 公众号 / RSS / 论文相关程序位置
- 确认每个程序的输入、输出、依赖、运行方式

输入：

- 参考代码目录
- 现有脚本说明

输出：

- 一份参考代码盘点清单

完成标准：

- 明确哪些代码可直接复用
- 明确哪些代码只能参考思路
- 明确哪些代码需要包一层适配器

#### Task 1.2 做接入决策

目标：

- 为每个来源决定“复用 / 适配 / 重写”

输出：

- 一份接入决策表

完成标准：

- X 的接入策略明确
- 公众号的接入策略明确
- arXiv 的接入策略明确
- 后续网页、新闻、视频类来源的扩展方式明确
- 没有模糊的“以后再说”路径

### 任务组 2：项目骨架与目录约定

#### Task 2.1 定义实现目录

目标：

- 在不写具体业务代码前，先确定实现目录边界

建议目录：

- `src/followhub/config/`
- `src/followhub/sources/`
- `src/followhub/models/`
- `src/followhub/pipeline/`
- `src/followhub/rendering/`
- `src/followhub/runtime/`
- `src/followhub/storage/`
- `tests/`

输出：

- 一份实现目录约定

完成标准：

- 每个目录责任明确
- 后续任务能映射到目录

#### Task 2.2 定义输入输出目录

目标：

- 确定原始数据、中间数据、页面输出、状态记录的落盘位置

建议目录：

- `data/raw/`
- `data/normalized/`
- `data/digests/`
- `data/state/`
- `site/follow/`

完成标准：

- 每类产物有固定位置
- 可支撑补跑和重建

### 任务组 3：配置系统

#### Task 3.1 定义配置文件结构

目标：

- 明确 sources、models、runtime、output 的配置结构

必须覆盖：

- X 账号
- 公众号源
- arXiv 分类/查询
- 通用网页来源
- 其他来源类型的扩展字段
- 最大补跑天数
- 模型任务绑定
- 输出目录

输出：

- 配置格式草案
- 示例配置文件

完成标准：

- 一份配置足以驱动单日任务

#### Task 3.2 定义配置校验规则

目标：

- 明确哪些配置是必填
- 明确错误时如何提示

完成标准：

- 用户能快速发现配置错误

### 任务组 4：数据模型与中间产物

#### Task 4.1 定义原始条目模型

目标：

- 统一多类来源的基础字段

必须包含：

- 来源类型
- 来源分类
- 来源名
- 作者
- 发布时间
- 标题
- 原文链接
- 原始文本或摘要
- 资源信息

完成标准：

- 当前核心来源都能映射到该模型
- 后续网页、新闻、视频来源也能在不推翻模型的前提下接入

#### Task 4.2 定义理解结果模型

目标：

- 定义摘要、相关性、重要性、推荐理由、翻译链接等字段

完成标准：

- 模型足以支撑日报页面渲染

#### Task 4.3 定义日报结果模型

目标：

- 定义单日摘要结果的落盘格式

完成标准：

- 可以支持重建页面、更新索引、补跑排错

### 任务组 5：信息源接入

#### Task 5.1 打通 X 最小接入链路

目标：

- 先接入一个最小可用的 X 数据来源

优先路径：

- RSS
- 现有监控程序

完成标准：

- 能拉到至少一组结构化条目

#### Task 5.2 打通公众号最小接入链路

目标：

- 获取至少一个公众号源的结构化内容

完成标准：

- 能输出统一条目模型

#### Task 5.3 打通 arXiv 最小接入链路

目标：

- 按分类或查询获取论文元数据

完成标准：

- 能输出标题、摘要、作者、时间、链接

#### Task 5.4 做来源适配层统一接口

目标：

- 不让不同来源直接耦合到后续 pipeline

完成标准：

- 三类来源输出接口统一
- 接口可以容纳后续网站、新闻、视频类来源

#### Task 5.5 定义扩展来源适配规范

目标：

- 为未来新增来源提供统一接入约束

至少覆盖：

- 个人网站 / 博客
- 公司网页
- 新闻站点
- 视频平台

完成标准：

- 新来源只需增加适配器，不需要重写 pipeline

### 任务组 6：摘要与筛选流程

#### Task 6.1 建立规则初筛

目标：

- 先用来源权重、关键词、条目特征做基础初筛

完成标准：

- 输出候选重点列表与保留线索列表

#### Task 6.2 建立 LLM 摘要流程

目标：

- 对候选内容生成短摘要

完成标准：

- 摘要可直接用于日报页面

#### Task 6.3 建立 LLM 价值判断流程

目标：

- 输出重要性、相关性和推荐理由

完成标准：

- 可以区分“重点”与“全量”

#### Task 6.4 建立论文专项解释流程

目标：

- 为论文生成“解决什么问题 / 用了什么方法 / 为什么值得看”

完成标准：

- 论文卡片可直接展示

### 任务组 7：页面渲染与输出

#### Task 7.1 定义日报页模板

目标：

- 定义页头、重点区、分类区、论文区、全量区

完成标准：

- 模板结构与 PRD 一致

#### Task 7.2 定义索引页与归档页模板

目标：

- 让日报可长期归档和浏览

完成标准：

- 能列出最近日报和历史归档

#### Task 7.3 建立静态输出约定

目标：

- 确定输出路径和文件命名

完成标准：

- 与未来 `tenstep.top/follow/` 的路径兼容

### 任务组 8：调度、状态、补跑与运行

#### Task 8.1 定义任务状态文件

目标：

- 记录上次成功日期、各阶段状态、错误信息

完成标准：

- 能支持补跑和排错

#### Task 8.2 定义单日执行流程

目标：

- 把采集、理解、渲染串成单日闭环

完成标准：

- 单日任务边界清晰

#### Task 8.3 定义补跑逻辑

目标：

- 启动时检查缺失日期并补齐

完成标准：

- 补跑窗口受配置控制

#### Task 8.4 定义运行入口

目标：

- 支持跑今天、补跑 N 天、重跑指定日期、只重建页面等入口

完成标准：

- 运维动作明确，不需要人工拼脚本

## 6. 关键里程碑

### 里程碑 A：参考代码与边界稳定

完成条件：

- 参考代码盘点完成
- 每类来源的接入策略确定
- 实现目录和数据落盘目录确定

### 里程碑 B：单日链路跑通

完成条件：

- 至少一个来源能够完成采集 -> 理解 -> 渲染 -> 输出

### 里程碑 C：三类来源统一输出

完成条件：

- X、公众号、arXiv 都能进入统一日报结构

### 里程碑 D：调度与补跑成立

完成条件：

- 支持跑今天
- 支持补跑最近 N 天
- 支持重建页面

## 7. 当前最应该先做的事

你已经明确“这里有参考代码”，所以当前最优先不是继续抽象，而是先完成两件事：

1. 给出参考代码位置
2. 对照 Task 1.1 和 Task 1.2，完成参考代码盘点与接入决策

在这两步完成前，不建议直接进入具体开发，否则很容易把目录、接口和已有实现做成两套体系。
